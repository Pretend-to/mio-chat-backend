# 上下文压缩机制 (Context Compression)

## 1. 背景
大模型在处理超长会话时，由于注意力机制和 Token 成本的限制，无法无限制地携带原始历史。业界常见的方案包括：
*   **摘要滚动 (Summary Rolling)**：将旧对话总结为一段话，放在新对话前。缺点是容易丢失关键细节。
*   **向量检索 (RAG)**：将历史存入数据库，按需检索。缺点是丢失了对话的时间线和连贯性。

Mio-Chat 采用的是 **“XML 结构化压缩”** 方案。

## 2. 核心设计

### 2.1 结构化分区 (XML Schema)
不同于普通的纯文本摘要，Mio-Chat 将压缩后的上下文划分为 5 个逻辑分区，这种设计仿照了计算机系统的“寄存器”概念：

*   **`<long_term_profile>`**: 长期记忆（用户画像、持续性偏好）。
*   **`<short_term_goals>`**: 任务目标。
*   **`<current_plan>`**: 待办清单（动态更新）。
*   **`<file_architecture_delta>`**: 文件变更摘要。
*   **`<constraints>`**: 核心约束逻辑。

这种结构化表达能有效降低 LLM 的“幻觉率”，并强制其遵循工程规范。

### 2.2 轮次边界保护
在压缩执行时，系统会智能识别对话轮次。如果 AI 正在执行工具（`tool_calls`）或尚未完成异步任务，系统会**延迟压缩**，确保逻辑链条的完整性，避免切断正在运行的任务。

### 2.3 极致的 Prompt Cache 命中
通过固定压缩块在消息链中的位置（通常置于头部），系统能最大化触发各大模型厂商（DeepSeek, OpenAI 等）提供的 **Context Caching (前缀缓存)** 性能。
*   **成本降低**：缓存命中后的 Token 价格通常只有原价的 1/10。
*   **延迟降低**：首字响应时间（TTFT）大幅缩短。

## 3. 触发机制
系统监测 `total_tokens` 水位线。当会话长度超过设定阈值（如 8,000 Tokens）时，系统会自动启动一个独立的后台任务执行压缩逻辑，并将压缩后的“晶体块”替换旧的历史消息。
