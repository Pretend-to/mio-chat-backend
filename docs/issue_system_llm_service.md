# Issue/Proposal: 建立后端系统级 LLM 服务通道与自动化任务机制

## 1. 背景与现状分析

目前 MioChat 的一些系统级维护任务（如：**对话标题自动总结**）主要由前端驱动：
- **逻辑分散**：前端 `contactor.js` 需要维护消息计数的触发逻辑。
- **职责不清**：前端 `openai.js` 适配器需要硬编码总结用的 Prompt，并自行调用通用的 `completions` 接口进行“Mock 请求”。
- **不可靠性**：如果前端刷新或未在特定计数值停留，可能会漏掉总结时机；且前端直接调用接口增加了通信成本和复杂度。

## 2. 建议改进方案

建议在后端引入 **“系统默认 LLM 渠道 (System Default Channel)”** 概念，并将日常维护功能收归后端管理。

### 2.1 后端配置增强
- **系统默认渠道**：在系统设置中允许显式指定一个“系统任务专用渠道/模型”（默认可指向 LLM 列表的第一项）。
- **任务策略配置**：后端统一配置总结触发的频率（如：第 2 条消息，之后每 6 条）。

### 2.2 逻辑下沉与自动化
- **后端自动总结**：
    - 后端在处理 `completions` 请求并检测到消息保存成功后，自动判断是否达到总结阈值。
    - 若达到，由后端直接调用系统 LLM 渠道进行总结。
- **主动推送机制**：
    - 总结完成后，后端通过 WebSocket 向该会话的所有客户端推送 `chat_title_updated` 消息。
    - 前端只需监听该事件并更新 UI，无需关心 Prompt 构造和模型调用逻辑。

### 2.3 接口标准化
- **专用 API**：如果前端需要手动触发（例如用户点击“重新生成标题”），后端应提供标准化的 `/api/chat/summarize` 接口，而不是让前端去拼接对话历史并发起完整的 LLM 请求。

### 2.4 自动化任务与定时器 (Automation & Cron)
- **任务管理中心**：在后端建立统一的任务调度中心，支持通过 UI 进行 CRUD 操作。
- **任务类型支持**：
    - **周期性任务 (Cron)**：如“每天早上 8 点生成昨日工作简报”。
    - **延时任务 (Delayed)**：如“10 分钟后提醒我查看某项任务进度”。
    - **事件驱动任务**：如“当会话达到 100 条消息时，自动进行长文本摘要压缩”。
- **执行环境**：任务在服务器后台运行，使用系统默认渠道，执行结果可通过 WebSocket 推送给用户或保存到指定会话。

## 3. 预期收益

1. **前端瘦身**：彻底移除前端硬编码的系统级 Prompt 和总结触发逻辑。
2. **架构解耦**：前端不再需要了解如何“总结对话”，只需负责展示后端给出的“结果”。
3. **一致性**：多端登录时，标题生成由后端统一处理，保证各端看到的标题变更同步且一致。
4. **全天候自动化**：系统任务不再依赖前端网页的开启，关闭浏览器后后端依然可以执行定时 AI 任务。
5. **易于扩展**：未来可以轻松增加其他后端自动任务，如：
    - **自动打标签 (Auto-tagging)**
    - **长对话自动压缩/摘要**
    - **情感分析与合规性检查**
    - **定时天气/新闻推送等 Agent 任务**
